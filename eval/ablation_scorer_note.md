# 消融实验（真轨迹 · credibility_scorer 工具效用验证）

- **变量**：credibility_scorer 工具调用（有 / 无），其余（任务集 domain-task、模型 deepseek-v4-flash、系统提示）固定
- **方法**：同一任务跑两遍，唯一差异是工具注册表中是否包含 credibility_scorer
- **日期**：2026-07-14

---

## 对比分析

| 指标 | 有 scorer | 无 scorer | 差异 |
|------|-----------|-----------|------|
| 主张验证权重 | 65 ✅ | 50 ❌ | 偏差 15pp |
| 主张验证得分 | 0.35（程序化） | 0.2（LLM估算） | 偏差 43% |
| 评分维度固定 | 三维度名称+权重由代码写死，不会偏离 ✅ | '域名权威性'被 LLM 编造为'来源权威性'，且未算出该维度得分 | 无 scorer 时 LLM 凭记忆编造维度名，名称和分数都不可控 |
| 综合得分 | 0.42 | 0.35 | 偏差 17% |
| 评分明细结构化 | ✅ 有 | ❌ 无 | 不可程序化验证 |

## 归因

- **无 credibility_scorer 时**：Agent 不知道评分维度的正确定义（权重比例、维度名称），只能凭模型记忆'猜'——导致权重错误（50% 而非 65%）、维度命名错误（'来源权威性' 而非 '域名权威性'）、分数 LLM 估算（而非程序化计算）
- **有 credibility_scorer 时**：评分维度、权重、计算方法全部由代码硬控，Agent 只需传 verdicts + page_metadata，scorer 返回精确的结构化分数——结果稳定、可审计
- credibility_scorer 的核心价值不是「算个分」，而是**保证评分的正确性（权重准确）、稳定性高（同输入同输出）和结构化（可程序化验证）**

## 局限

- 单次运行，未取均值；D4 起应每组多次运行取均值再下结论
- 只测了一个任务（domain-task），需扩展到更多任务类型
- 无 scorer 时 Agent 仍尝试手动估算分数（因为 SKILL.md 要求输出评分），实际场景中可能完全跳过评分环节
