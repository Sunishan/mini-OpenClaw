# mini-OpenClaw 信息核查 Agent

本项目实现了一个面向网页可信度判断的信息核查 Agent。用户给出网页链接或新闻内容后，Agent 会自动读取页面、提取核心事实主张、搜索外部证据、进行交叉验证，并输出结构化可信度结论。

项目不是简单地给出“可信 / 不可信”的判断，而是尽量还原核查依据：哪些主张被证据支持，哪些主张无法验证，哪些地方存在来源、时效性或技术风险。

## 核心功能

### 1. 网页内容读取

Agent 会优先使用内置网页读取工具获取页面标题、正文、作者、发布时间、域名等信息。

对于普通 HTTP 抓取失败、SSL 证书异常、页面动态渲染或正文抽取困难的网页，系统可以继续尝试：

- Playwright MCP：通过浏览器方式访问动态页面
- Firecrawl MCP：在必要时进行更强的网页正文抽取

### 2. 核心主张提取

系统会从网页正文中提取可验证的事实性主张，例如：

- 事件是否发生
- 事件发生时间
- 涉及地点和主体
- 数值、等级、预警、统计数据
- 因果关系或趋势判断

早期的正则式 `claim_extractor` 只能做有限候选句提取，容易漏掉复杂表达。当前设计把完整主张提取流程放进 Skill，由大模型根据任务语境直接抽取“原子主张”，再交给后续验证步骤处理。

### 3. 外部证据搜索

主张提取后，Agent 会搜索外部来源进行验证。优先级通常是：

1. 官方来源，例如政府、气象台、监管机构、原始公告
2. 权威媒体或专业机构
3. 多个独立第三方报道
4. 普通转载、聚合页或低可信来源

对于新闻类、灾害类、政策类、金融类等时效性强的信息，Agent 会特别关注发布时间和事件发生时间，避免用过期证据验证当前事件。

### 4. 交叉验证

Agent 会把网页中的每个核心主张和外部证据逐项比对，形成以下判断：

- `supported`：有可靠证据支持
- `contradicted`：被可靠证据反驳
- `unverifiable`：证据不足，暂时无法确认
- `partially_supported`：部分内容被支持，但存在缺口或不确定性

这样可以避免把整篇文章一次性粗暴判断为真或假，而是说明哪些部分可靠、哪些部分存疑。

### 5. 可信度报告

最终输出会包含：

- 原网页基本信息
- 核心主张列表
- 每条主张的验证结果
- 主要证据来源
- 可疑点和风险点
- 综合可信度结论
- 对事件的简要还原

报告目标是让用户知道“为什么可信”或“为什么不可信”，而不是只给一个结论。

## 实现流程

整体流程如下：

```text
用户输入网页链接或新闻内容
  ↓
Agent 判断任务属于信息核查
  ↓
加载 information-verification Skill
  ↓
读取网页正文和元信息
  ↓
提取可验证的原子主张
  ↓
搜索外部证据
  ↓
对主张和证据进行交叉验证
  ↓
计算可信度并整理风险点
  ↓
生成结构化核查报告
```

## 关键模块

### Agent 主循环

主循环位于：

```text
agent/loop.py
```

它负责执行 ReAct 流程：

```text
模型思考
  ↓
决定是否调用工具
  ↓
执行工具
  ↓
把工具结果返回给模型
  ↓
继续推理或输出最终回答
```

信息核查不是硬编码在主循环中，而是作为一种可加载能力由 Skill 指导模型完成。

### 信息核查 Skill

核心 Skill 位于：

```text
skills/information-verification/SKILL.md
```

它定义了信息核查任务的完整工作流，包括：

- 什么时候进入信息核查模式
- 如何读取网页
- 如何提取事实主张
- 如何选择证据来源
- 如何处理 SSL 失败、动态页面、搜索失败等问题
- 如何组织最终报告

把流程放进 Skill 的好处是：主循环保持通用，信息核查流程可以独立迭代，不需要把大量领域逻辑写死在 `agent/loop.py` 或 `agent/prompts.py` 中。

### 网页可信度工具

相关工具位于：

```text
tools/credibility/
```

主要包括：

- `webpage_reader.py`：读取网页正文和元信息
- `claim_extractor.py`：早期主张提取辅助工具，目前更偏候选句 / legacy 角色
- `kb_retriever.py`：检索外部证据
- `cross_validator.py`：比对主张和证据
- `credibility_scorer.py`：计算可信度
- `report_generator.py`：生成报告

这些工具为信息核查提供结构化能力，但具体调用顺序由 Agent 根据 Skill 和任务上下文决定。

### MCP 网页读取补充

MCP 客户端位于：

```text
mcp/client.py
```

信息核查中用到的外部 MCP 主要有：

- Playwright MCP：用于访问动态网页、处理浏览器渲染页面
- Firecrawl MCP：用于普通抓取失败时抽取正文

这两个 MCP 并不替代内置工具，而是在内置抓取失败或内容不完整时作为补充手段，用于提升复杂网页的读取成功率。

## 示例：网页可信度判断

用户输入：

```text
判断这个网页的可信度：https://example.com/news
```

Agent 的实际处理过程大致是：

1. 识别这是信息核查任务
2. 加载 `information-verification` Skill
3. 读取目标网页正文
4. 如果网页读取失败，尝试浏览器或 Firecrawl 方式
5. 提取文章中的核心事实主张
6. 搜索官方来源和第三方证据
7. 对每条主张进行支持、反驳或不可验证判断
8. 汇总网页来源、内容一致性、证据质量、可疑点
9. 输出最终可信度报告

## 设计特点

- 流程可解释：不仅输出结论，还展示主张、证据和判断依据
- 工具可扩展：内置工具和 MCP 工具统一注册到工具系统
- Skill 驱动：复杂核查流程由 Skill 描述，主循环保持通用
- 动态网页兼容：普通抓取失败时可以借助 Playwright / Firecrawl

## 当前重点

当前项目主要围绕“网页可信度判断”能力展开。相比普通聊天机器人，本项目更强调：

- 对网页原文的读取能力
- 对事实主张的拆解能力
- 对外部证据的检索能力
- 对证据和主张的一致性判断
- 对最终结论的结构化解释
